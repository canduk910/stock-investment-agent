# chat/ — LLM 계층 (설명만 하는 챗봇)

이 계층의 존재 이유: **판정·숫자는 코드가 확정하고, LLM은 그 결과를 설명만 한다.** 상세 규칙은 `llm-safety-guide` 스킬.

- **모델은 `CHAT_MODEL` 단일 출처**(`chat/tools.py`, 현재 `"gpt-5.6-luna"`). 코드 어디에도 모델 문자열을 산재시키지 않는다 — 바꿀 땐 이 상수만.
- **★모델별 필수 파라미터 = `CHAT_MODEL_PARAMS`(단일 출처, 매 `create()`에 병합)**. `gpt-5.6-luna`는 **추론형**이라 chat/completions에서 **function tools를 쓰려면 `reasoning_effort='none'`가 필수**(미지정 시 400 `Function tools with reasoning_effort are not supported`). 또 이 계열은 구형 `max_tokens`/`temperature`를 안 받는다(앱은 안 넘김). `chat.py::_create_with_retry`가 `{**CHAT_MODEL_PARAMS, **kwargs}`로 병합(스트리밍 포함), `report.py`·`intent_gen.py`도 `**CHAT_MODEL_PARAMS` 전달. **비추론형 모델(gpt-4o 등)로 바꾸면 `CHAT_MODEL_PARAMS={}`로 비운다.**
- **위험 차단 = 2층 방어**(오탐 과다 완화). ①**결정적 키워드 정규식(`guardrail_label`)** = 하드블록(안전 하한선) — 단정예측/내부정보/시세조종/과도위험 4유형은 **LLM 미호출**로 즉시 차단(거절이 아니라 위험 환기 + 분산 안내로 방향 전환). ②정규식 미매치인데 **ML이 risk_guardrail로 분류**하면(오탐 가능) `chat.py::_reclassify_risk`가 **CHAT_MODEL로 2차 재분류**(`{block:bool}`) — 오탐이면 **정상 답변으로 진행(구제)**, 위험 확정이면 차단. 재분류 전체 실패는 재시도 후 **보수적 차단**(안전 우선, 드묾). LLM은 '차단 대상 여부'만 분류(자문·설명은 본 답변 LLM). 차단 판단은 `chat()`·`chat_stream()` 공통. **api/chat 라우트의 `live_judgement` 스킵은 `guardrail_label`(결정적)일 때만** — ML-risk는 재분류로 허용될 수 있어 국면 필요. **잔고·손익·포트폴리오 조회가 risk로 오분류돼 하드블록되던 문제를 이 2층 구조 + 데이터셋 보강으로 해소**(실측: "내 잔고 확인해줄래?" 개선 전 차단 → 후 정상).
- **인텐트 6분류 = ML 사전분류.** `intent_gen.py`가 CHAT_MODEL로 [질문→라벨] 균형 데이터(`data/intent_dataset.tsv`)를 생성 → `intent_train.py`가 `TfidfVectorizer(char_wb,(2,4))+LogisticRegression`으로 학습 → `chat/models/intent_clf.joblib`. 한글은 형태소분석기 없이 char n-gram으로 처리. 재생성/재학습해도 파이프라인 정의는 불변. **ML 정확도는 비결정적** → 단위 테스트는 인터페이스·가드레일만 고정하고 정확도는 단정하지 않는다. **단, 정당 조회의 risk 오탐은 안전 이슈**(하드블록) → `test_intent.py`가 `data/intent_dataset.tsv`를 즉석 학습해 큐레이션 정당 질의(잔고·손익·포트폴리오·추가편입)가 `!= risk_guardrail`임을 검증(데이터셋 품질, tfidf+lbfgs 결정적). 오분류 표현은 정답 라벨(portfolio_advice 등)로 데이터셋 보강 후 재학습(`uv run python -m chat.intent_train`, 모델 커밋). 비차단 라벨끼리(general_qa↔stock_analysis) 오분류는 안전 무관(LLM이 tool_choice로 결정).
- **인텐트 → 네비게이션 패널 결정적 라우팅(최초 설계 반영)** — `chat/intent_panel.py`: **인자 없는 3개 라벨**만 패널로 매핑(`macro_view→show_macro_dashboard`·`watchlist_mgmt→show_watchlist`·`portfolio_advice→show_balance`). `chat()`·`chat_stream()`이 `intent=classify()` 1회 계산 후 `merge_intent_panel(intent, popups)`로 **인텐트 패널을 popups 맨 앞에 주입(+LLM 중복 dedup)** → 프론트가 `popups[0]`로 전환하므로 **인텐트가 패널을 권위적으로 결정**(폴백에도 적용). ticker 필요한 종목리포트(`stock_analysis`/`analyst_report`)·args 필요한 편집(`manage_watchlist`)은 인텐트가 인자를 못 뽑아 **LLM function calling 병행**(매핑 0). **안전**: 편집·매매 패널은 절대 매핑 안 함(오분류가 편집/주문 유발 0)·가드레일 선차단 불변·팝업 계약/`popupRouter` SSOT 재사용(프론트 변경 0). 값(툴명)이 표시 툴·CONTENT_TOOLS 아님을 테스트로 잠금.
- **포트폴리오 상담 활성(코드 근거 자문)**: `build_prompt` 블록 ⑦ [포트폴리오 상담] — 잔고 스냅샷 근거 집중도·비중·손익 분석 + 국면 권장 현금비중 대비 조정 방향 + **추가편입 후보**(관심종목·분산·국면 관점, 근거 동반) + 국면·분산 부족 특성/**새 종목 아이디어**(방향 제안, 특정 종목 단정·구체 수치 날조 금지). ④/⑦ 완화로 후보를 actionable 제시 허용. **국면은 현금비중만 관리**(항목3 — `single_cap`/`per_max`/`pbr_max` 게이트 폐기, 종목별 진입 커트로 판정하지 않음). **불변**: ⑤ 단정 금지·**내부정보/시세조종 거부·수익 보장 금지**(재분류 backstop)·면책·과열 국면(현금비중 높음) 규율.
- **목표가 추천(에이전트) + 매수/매도 분리**: `manage_watchlist` set_target 이 `target_price`(매수)·`sell_target_price`(매도) 둘 다 수용(최소 1개, 프론트 popupRouter 검증). `build_prompt` **⑨ [목표가 추천]** 블록 — 사용자가 목표가 추천을 원하면 **애널리스트 목표주가 우선 인용("리포트에 따르면")→52주 고저·현재가 위치→현재/자기평균 PER** 순으로 **매수/매도 참고 범위**(X~Y원) 제시 후, 반영 원하면 `manage_watchlist(set_target, target_price, sell_target_price)`로 **제안**(확인 카드에서 사용자 [확인]—자동 저장·자동 매매 없음). **불변**: 단일 확정수치·"반드시 오른다" 금지, 스냅샷/리포트 인용만(수치 날조 금지), 상태 판정(reached/near/far)은 코드, 면책. `view_context._stock_context` 가 추천 근거(애널리스트 목표주가·**52주 고/저 원값**·PER)를 스냅샷에 싣고, `_watchlist_context` 는 **매수·매도 목표가/상태 둘 다** 인용.
- **기준표는 자동 생성**(`build_prompt.py::build_criteria_text`). `THRESHOLDS`·`INDICATOR_LABELS`·`VIX_PANIC`을 import해 만든다 — 임계값 숫자를 프롬프트에 하드코딩하면 3중 일관성이 깨진다. 시스템 프롬프트 필수 6블록(역할·판정출처고정·기준표+judgement·REGIME_PARAMS·설명지침·팝업규칙)과 면책 고지 포함. `judgement`는 매 호출 최신값 주입.
- **세션은 서버 인메모리**(`session.py`의 `SESSIONS[session_id]`, 슬라이딩 윈도우 8). 시스템 프롬프트는 매 호출 재주입(누적 X → 국면 변경 자동 반영), tool 메시지는 히스토리에 미누적. 휘발성(서버 재시작 시 손실 — 클라우드 전환 시 DynamoDB/Redis).
- **팝업은 tool_calls에서만** 생성된다(`chat.py`). LLM은 "무엇을 띄울지"만 결정하고 실데이터는 프론트가 직접 조회(환각 차단). tool 결과는 `{"ok":True}` 확인만 되먹인다. 반환 계약 `{text, popups:[{name,args}]}`. `chat.py`는 tool_call을 **범용 포워딩**(name/args를 그대로 popups로) — 새 툴을 `tools.py`에 추가해도 `chat.py`는 무변경. 툴 정의(`tools.py`)는 `show_stock_report`·`show_macro_dashboard`·`show_watchlist`·`manage_watchlist`·**`show_balance`**(UX 개편, 무파라미터). `show_balance`는 계좌 잔고·평가액·수익 **현황**을 물을 때만 호출하고, **리밸런싱·분산 조언은 팝업 없이 텍스트로만**(데이터는 프론트가 `/api/balance` 조회, 판정/조언은 LLM 서술) — `build_prompt.py` ⑦블록 규칙과 짝. 프론트 라우팅 계약은 `frontend/src/lib/popupRouter.js`(`POPUP_KIND` 5종 SSOT).
- **★콘텐츠 툴(표시 툴과 구분)**: `tools.py::CONTENT_TOOLS`(집합) + `run_content_tool`(실행 레지스트리)이 단일 출처. 표시 툴(`show_*`)은 `{ok:True}`만 되먹이고 실데이터는 프론트가 조회(환각 차단)하지만, **콘텐츠 툴은 서버가 실행해 실제 텍스트를 되먹여 LLM이 요약**한다. `chat.py`(`chat()`·`chat_stream()` 둘 다)가 `CONTENT_TOOLS`로 분기 — 콘텐츠 툴은 **popups에서 제외**(팝업 아님)하되 assistant tool_calls·tool 메시지는 전부 짝을 맞춘다(`call_{i}` 일치). 현재 콘텐츠 툴: **`summarize_youtube`**(`video_url` → `collectors.youtube.fetch_transcript` 자막 → LLM 요약) · **`search_report`**(`query` → `rag.store.search_reports` **업로드 PDF(reports/ 폴더) RAG** top-k 청크) · **`fetch_analyst_reports`**(`ticker` → `analyst_service.fetch_and_summarize_for_ticker` **네이버 애널리스트 리포트 신규 수집**·저장된 요약 되먹임). **안전(build_prompt ⑧블록)**: 외부 콘텐츠는 "영상/리포트에 따르면"으로 **출처 귀속** 요약, 에이전트 자체 매수/매도 판정 금지, 면책 유지. `fetch_transcript`는 **타임아웃(12s) 상한**(youtube-transcript-api 자체 타임아웃 없음 → hang 방지)·실패/부재는 graceful None.
- **★두 리포트 채널 구분 + "없으면 먼저 제안"(사용자 결정)**: 애널리스트 리포트가 두 경로로 나뉜다 — (i) `search_report`=사용자가 **직접 올린** PDF RAG, (ii) `fetch_analyst_reports`=**네이버 수집**. 과거엔 챗에 수집 툴이 없어 "확보해줘"가 `search_report`로 떨어져 "PDF 폴더 재인덱스" **오답**을 냈다. 이제 build_prompt ⑧이 두 채널을 명시하고, **저장된 리포트가 없으면 목표주가를 지어내지 말고 먼저 "지금 수집해 올까요?"로 제안**(자동수집 아님 — 수집이 10~50초로 느려 모든 종목 질문마다 자동 트리거하지 않는다는 기존 설계와 일치) → 사용자가 동의/"확보·가져와·수집"하면 `fetch_analyst_reports(ticker)` 호출. 툴은 `_looks_like_ticker`(6자리)로 불량 티커 시 수집 미호출(불필요 크롤 방지)·`ANALYST_CHAT_FETCH_LIMIT=5`로 지연 캡·0건 graceful. `chat.py`는 무변경(CONTENT_TOOLS 범용 분기). 라이브 e2e: "LG이노텍(011070) 리포트 확보해줘"→5건 수집→증권사별 목표주가(대신·유안타·DS·iM·교보) 출처 귀속+면책.
- **증권사 리포트 RAG(`rag/` 패키지)**: `reports/` 폴더 PDF → `ingest`(pdfplumber 추출 + 표보존 청킹) → `embed`(OpenAI text-embedding-3-small) → `store`(**numpy 코사인 top-k**, FAISS 대신 dep 절감 · `.cache` 영속 · 싱글톤). 관리 API `POST /api/reports/reindex`·`GET /api/reports/status`(api/reports.py). 리포트=정적 문서라 인덱스 캐시 허용(현재가 아님, 원칙1 무관). `reports/`는 gitignore(저작물·사용자 데이터). 스캔/이미지 PDF vision-OCR 폴백은 Phase 2b(미구현 — pdfplumber 텍스트 PDF 우선).
- **애널리스트 리포트 요약·상담 연계(네이버)**: `collectors.naver_research`(수집) → `analyst_report.summarize_report(text, meta)`(chat/report.py 패턴 재사용 — `AnalystReportSummary`(Pydantic) 안전강제[핵심요지·리스크 min1·면책 필수, 목표주가 optional]·생성·검증·1회 재시도·폴백) → `analyst_store`(`.cache/analyst_reports.json`, **ticker 키·`report_id`(nid) idempotent** — 중복 skip) → `analyst_service.fetch_and_summarize`(수집→다운로드→요약→저장 **ThreadPool 병렬**, 항상 카운트 dict). **핵심 안전**: 이 요약은 **해당 증권사 리포트의 내용 인용**(투자의견은 '리포트가 밝힌 의견'으로 귀속, Literal 로 가두지 않음)이지 **에이전트 자체 매수/매도 판정이 아니다** — 출처 귀속·면책을 프롬프트+표시로 강제. 라이브 e2e 검증됨(GS건설: 다운로드→pdfplumber 추출→요약→검증 통과).
- **시황(매크로) 챗 상담 + 금일의 요약(애널리스트 패턴 미러)**: ①`market_outlook.format_market_outlook_context(entry)` = 저장 시황 entry→세션 핀 컨텍스트 텍스트(애널리스트 `format_report_context` 대응 — 시장 전체라 종목·목표주가 없음·**시장전망** 있음). `market_outlook_store.get(report_id)`(MARKET_SCOPE 단건). `POST /api/chat/market-outlook-context`가 **같은 `Session.report_context` 슬롯**에 핀(애널리스트와 공유 — 한 번에 하나). ②`market_outlook_combined.summarize_recent_outlooks(limit=5)`(`analyst_combined` 미러) — 저장 요약 최근 **5개**만으로 LLM 1회 종합·**중복제거**(PDF 재다운로드 0). `CombinedMarketOutlookSummary`(`market_outlook_combined_schema.py`): `종합요약`(min1·**max10**)·`시장전망분포`(리포트 전망 집계 "중립 3·신중 2")·`면책고지` 필수. **`report_count`는 코드 계산**(LLM 환각 방지)·0개 LLM미호출·검증 실패 1회 재시도→폴백. **함정(라이브 실증)**: 저장 `시장전망`이 **긴 문단**(짧은 라벨 아님)이면 LLM 이 `시장전망분포`를 문자열 대신 **dict**로 내 검증 실패 → 프롬프트에 "하나의 문자열·객체 아님" 명시 + 스키마 `field_validator(mode="before")`가 dict/list→컴팩트 문자열 강제(방어). (애널리스트는 `투자의견`이 짧아 이 문제 없음.) **안전**: 여러 시황 리포트 **종합·인용**(시장 판정 아님)·복수 출처 귀속·면책 강제.

- **현재 보는 화면 컨텍스트(잔고·관심종목·종목상세)**: 표시 툴은 `{ok:True}`만 되먹여 LLM이 화면 숫자를 몰랐다 → **서버가 조회한 스냅샷을 세션에 실어** 대화에 반영. ①`Session.view_context`(핀 슬롯, report_context와 별개·단일 slot=현재 보는 화면). ②`chat/view_context.py::build_view_context(kind, args)`(`DATA_BEARING_KINDS={balance,watchlist,stock_report}` SSOT) — 재사용 함수로 **서버 재조회**(잔고 `inquire_balance`·관심종목 `build_watchlist_view`·종목 경량 `inquire_price`+`_pos_52w`+애널리스트 `list_reports` 0KIS), **컴팩트 텍스트**(첫 줄 `기준시각`·top-N[잔고8·관심10·애널2]·1500자 상한)·**절대 예외 없음**(실패는 "일시 조회 불가" 노트/None). ③`_build_system_prompt`가 report 블록 뒤로 `_VIEW_CONTEXT_HEADER`+본문 주입(숫자 인용만·날조 금지·스냅샷 시각 환기·판정 금지·면책). **두 경로로 핀**: (P1) 프론트가 패널 변경 시 `POST /api/chat/context`, (P2) 챗이 `show_balance/watchlist/stock_report` 호출 시 `chat.py::_display_tool_result`가 서버 스냅샷을 tool 결과 `{"ok":True,"summary":...}`로 되먹여 **같은 턴 즉답**(팝업은 유지). `tools.py::VIEW_CONTEXT_TOOLS`+`view_context_kind_args`가 툴↔kind 매퍼(CONTENT_TOOLS와 별개 — 뷰 툴은 popups 유지). **guardrail은 view_context 있어도 선차단 불변**(classify가 `_build_system_prompt`보다 앞). 라이브 e2e: "내 잔고 어때/손실 종목?"→show_balance→서버 조회→LLM이 실수치(골프존 -22.90%) 인용+기준시각+면책. **테스트 주의**: chat 유닛테스트는 `build_view_context`를 기본 stub(None)해야 실 FRED/KIS·`requests_cache` 전역설치로 fred/vix 오염 안 됨.
- **세션 리포트 상담 컨텍스트(Phase D)**: `Session.report_context`(핀 슬롯, **슬라이딩 윈도우와 별개** — 후속 여러 턴 참조)·`set/clear_report_context`. `chat.py::_build_system_prompt(judgement, session)`가 **양 경로(chat·chat_stream) 단일 출처**로 `build_prompt` + (컨텍스트 있으면)리포트 블록을 조립 — "출처 귀속('리포트에 따르면')·매수/매도 판정 금지·면책 유지" 재강조. **필수 6블록·가드레일은 불변**(guardrail 은 컨텍스트가 있어도 코드가 선차단, LLM 미호출). 엔드포인트 `POST /api/chat/report-context`가 store 에서 조회한 요약을 세팅(**요약 본문은 프론트가 신뢰전송하지 않음** — 서버가 조회, 환각/조작 차단).
- **스트리밍(`chat_stream`)**: SSE용 동기 제너레이터. 이벤트 `{type: "stage"|"token"|"popups"|"done"}`, stage enum = `analyze|regime|generate|summarize`. CHAT_MODEL `stream=True`. **스트리밍 tool_calls는 델타 조각을 index별로 이어붙여 재조립**(`_accumulate_tool_calls`) — 부분 도착하는 `function.name`/`arguments`를 누적해야 완전한 팝업이 나온다. 기존 동기 `chat()`은 그대로 유지(논스트림 폴백·기존 테스트). - **[P2] 구조화 리포트(§6.5b)**: `report_schema.py`의 `StockReport`(Pydantic)가 안전 요건을 **스키마 레벨에서 강제** — `리스크요인 min_length=1`(장밋빛 일변도 방지)·`면책고지` 필수·`종합의견 Literal["긍정적","중립","신중"]`(매수/매도 단정 라벨 원천 차단). `report.py::generate_stock_report`는 `CHAT_MODEL`로 JSON 생성 → 검증 실패 1회 재시도 → 재실패 시 **폴백**(정량요약만 + `validation_failed=True`, 부분실패 보존, §5.1). `report_store.py`가 `.cache/stock_reports.json`에 `(ticker, created_at)`·`regime_at_creation`으로 히스토리 누적(과거 평가 비교 데모). 테스트는 FakeOpenAI mock(라이브 미호출).
- **국면 = 현금비중만(build_prompt ④블록) — 진입게이트 지침 폐기(항목3)**: 국면별 종목 진입신호 지침(`single_cap`/`per_max`/`pbr_max` 게이트)은 "너무 보수적"이라 삭제됐다. ④블록은 "국면은 **현금비중만** 관리하며 개별 종목 PER/PBR 은 참고 데이터로만, 국면 상한 초과/진입 차단 커트는 없다"로 재작성. `_format_params` 는 `cash` 만 인용(3중 일관성 유지). 리포트의 `국면정합성` 은 LLM 이 국면명+권장 현금비중 스탠스 대비 '최종 적합성'을 서술(`report.py::_regime_block`).

## 대화기록·공동 리포트 DB (Phase 4~5)
- **대화기록**(유저별): `history_models`(Conversation/ChatMessage)·`history_store.HistoryStore`(대화 CRUD·`add_turn`·`recent_messages`[hydrate용]). 챗 라우트가 세션 hydrate + 턴 write-through(api/CLAUDE). `Session.hydrate(msgs)`가 슬라이딩 히스토리만 교체(핀 불변).
- **애널리스트·시황 요약 = 공동 DB**(전역 공유, user 무관): `report_models.AnalystReportRow`(scope_key=ticker|`__MARKET__`, `(scope,report_id)` 유니크)·`report_repo.ScopedReportRepo`(**세션 팩토리로 메서드마다 새 세션 → 병렬 수집 스레드 안전**). `analyst_store`·`market_outlook_store`는 thin SQL 어댑터(인터페이스 has/upsert/list/get 불변 → 라우트·service·view_context 무변경). `default_store()`는 앱 기본 세션팩토리, 테스트는 인메모리 sessionmaker 주입.
- **시황 3줄요약(`세줄요약`, 항목4)**: `MarketOutlookSummary`에 `세줄요약: list[NonEmptyStr]`(min1·max3) 필드 추가 — 프론트 컴팩트 카드 미리보기용 3줄 압축. `market_outlook.py::_build_summary_prompt`가 "리포트 시황 3줄 압축(내용 인용)"을 지시. **기존 저장 레코드는 `세줄요약` 없음**(재검증 안 함) → 프론트가 `핵심요지[:3]`로 폴백, 재수집 시 신규 레코드에 채워짐. 안전(리포트 인용·출처 귀속·면책·시장판정 아님) 불변.
