# 투자 에이전트 QA 검증 방법론

핵심 원칙: **존재 확인이 아니라 교차 비교.** "API가 있는가"가 아니라 "API의 응답이 소비자의 기대와 일치하는가"를 검증한다. 두 컴포넌트가 각각 올바르게 구현돼도 연결 지점의 계약이 어긋나면 런타임에 죽는다.

## 1. 안전 검증 (최우선 — 실패 시 즉시 리더 보고)

grep으로 기계적으로 잡을 수 있는 항목부터:

| 항목 | 검사 방법 | 통과 기준 |
|---|---|---|
| 주문 API 호출 없음 | `grep -rniE "order_cash|order_rvsecncl|buy_order|sell_order|주문" --include="*.py"` | 조회 외 주문 계열 함수/TR 0건 |
| API 키 하드코딩 없음 | `grep -rnE "(app_?key|secret|api_?key)\s*=\s*['\"][A-Za-z0-9]{10,}" --include="*.py"` | 0건 (환경변수 로드만 허용) |
| 단정 표현 없음 | 프롬프트 파일에서 `grep -n "반드시\|무조건\|확실히 오른"` | 금지 지시문 내 인용 제외 0건 |
| 현재가 캐시 금지 | 캐시 저장 코드에서 현재가/등락률 필드가 저장되는지 추적 | 시세 필드 캐시 저장 0건 |
| 실패 응답 캐시 미저장 | 캐시 `set` 호출 전 `partial_failure` 검사 존재 여부 | 검사 없이 저장하는 경로 0건 |
| 면책 고지 | 시스템 프롬프트 필수 블록 + 리포트 UI 하단 고정 | 양쪽 존재 |

grep은 스크리닝일 뿐이다 — 히트가 0이어도 호출 그래프를 한 번 눈으로 따라가 우회 경로(동적 함수명 등)가 없는지 확인한다.

## 2. 3중 일관성 검사 (이 프로젝트 고유의 핵심 검증)

임계값이 세 층에서 같은 값을 가리키는지:

```
① 코드 상수 (THRESHOLDS / REGIME_PARAMS / VIX_PANIC / VALUATION_BAND)
② build_criteria_text() 출력 — ①을 import해 생성하는지, 숫자를 직접 타이핑한 곳은 없는지
③ Pydantic 스키마 필드 범위 (P2 적용 시)
```

검증 방법: ② 프롬프트 조립 코드에서 숫자 리터럴을 grep(`grep -nE "[0-9]+\.?[0-9]*" chat/build_prompt.py`)해서, 상수 import가 아닌 하드코딩 숫자가 있으면 실패. 추가로 `build_criteria_text()`를 실제 실행해 출력에 상수 값이 반영되는지 확인한다.

**함정 체크**: 현금비중 관점 통일 — `CASH_RATIO`(주식비중 관점)와 `REGIME_PARAMS.cash`(현금비중 관점)가 공존하며 서로 다른 숫자를 갖고 있으면 실패. 한 상수만 남아야 한다.

## 3. 경계면 교차 비교 (양쪽 동시 읽기)

| # | 생산자 (왼쪽) | 소비자 (오른쪽) | 확인 사항 |
|---|---|---|---|
| 1 | `/bundle` 반환 dict의 키 | 프론트 훅/컴포넌트가 접근하는 필드 | 키 이름·중첩 구조·null 처리 일치 |
| 2 | `chat()` 반환 `{text, popups}` | 프론트 말풍선/팝업 분기 코드 | popups[i].name/args 파싱 일치 |
| 3 | TOOLS의 툴 이름·enum 값 | 프론트 팝업 라우팅 switch 키 | `show_stock_report` 등 3종 1:1 매핑, enum 값 대응 컴포넌트 존재 |
| 4 | `judge_regime()` 반환 dict | `build_prompt.py`가 읽는 필드 | regime/votes/key_drivers/override/missing_indicators |
| 5 | `build_stock_summary()` 반환 | 리포트 프롬프트 + 프론트 정량 카드 | 필드명·None 처리 양쪽 일치 |
| 6 | `partial_failure` 기록 로직 | 프론트 섹션 fallback | 실패 섹션명 문자열이 양쪽에서 동일 |
| 7 | 인텐트 분류 출력 라벨 | 라우팅 분기 | 6개 라벨 전부 분기 존재, 죽은 분기 없음 |

각 항목은 반드시 **양쪽 파일을 동시에 열어** 비교한다. 한쪽만 읽고 "맞겠지"로 통과시키지 않는다.

## 3.5 TDD 준수 검증

모든 구현 모듈은 `tdd-workflow` 스킬의 Definition of Done을 충족해야 한다:

1. **테스트 존재 + 직접 실행**: 전체 스위트를 실행한다 (`pytest`, 라이브 테스트는 `-m "not live"`로 제외). 테스트 없는 모듈은 자동 실패.
2. **경계 케이스 커버리지**: `quant-engine-rules` §5의 경계값 목록과 테스트 파일을 대조 — 누락된 경계는 실패 항목으로 기록.
3. **test-first 증거**: `_workspace/` 명세가 "테스트 목록 → 구현" 순서로 기록됐는지 확인.
4. **테스트 변형 흔적**: 임계값 상수·assertion이 구현에 맞춰 수정된 흔적(스펙과 다른 기대값)이 있으면 스펙 재확인을 요구한다.
5. **모킹 남용**: 경계(HTTP/OpenAI/캐시 저장소) 밖의 내부 로직까지 mock돼 실제 코드가 한 줄도 실행되지 않는 테스트는 무효로 판정.
6. **가짜 테스트 스크리닝**: assertion이 없거나 항상 참인 테스트(`assert True`, 결과 미사용)를 grep으로 확인.

## 4. 기능 스펙 검증

- 단위 테스트를 **직접 실행**한다 (`pytest` 등). 존재 확인은 통과가 아니다
- 매크로 엔진: VIX 오버라이드가 투표보다 먼저 검사되는지 코드 순서 확인
- 지표 누락 시 기본값으로 채우지 않고 `missing_indicators`로 명시하는지
- 워치리스트 진입 신호: 수축기 `single_cap=0` 게이트가 실제로 신호를 차단하는지

## 5. 리포트 형식

`_workspace/{week}_qa-inspector_report.md`:

```markdown
# QA 리포트 — {대상 모듈} ({날짜})
## 요약: 통과 N / 실패 M / 미검증 K
## 실패 항목
- [안전|경계면|일관성|스펙] {제목}
  - 위치: {파일}:{라인}
  - 현상 / 기대: ...
  - 수정 방향: ... → 담당: {agent}
## 미검증 (대상 코드 미존재 등 사유)
```

실패는 담당 에이전트에게 SendMessage로 즉시 전달하고, 경계면 이슈는 생산자·소비자 양쪽에 알린다. 검증은 각 모듈 완성 직후 즉시(incremental) — 전체 완성 후 몰아서 하면 경계면 버그가 후속 모듈에 전파된 뒤라 수정 비용이 커진다.
