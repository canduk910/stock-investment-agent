# Backend Architecture — 백엔드 아키텍처 거버넌스 · 리팩토링 방법론

이 스킬은 **backend-architect**가 백엔드를 "하나의 응집된 시스템"으로 유지하는 방법을 담는다. QA가 *동작*을 검증한다면, 여기서는 *구조*를 설계·점검한다. 핵심은 규칙 암기가 아니라 **왜 이 경계가 있는가**를 이해해 새 상황에서도 옳게 판단하는 것이다.

## 1. 왜 아키텍처 역할이 따로 필요한가

동작 검증(QA)만으로는 "지금은 맞지만 곧 무너질 구조"를 못 잡는다. 계약이 없어 매번 오타로 터지고, 상수가 두 곳에 있어 언젠가 어긋나고, 판정 로직이 LLM으로 새어 재현 불가능해진다. 이런 것은 테스트가 green이어도 **구조적 부채**다. 아키텍트는 이 부채를 선제적으로 진단하고, 재발하지 않게 구조를 바꾼다.

## 2. 계층 지도 (Layering) — 백엔드의 척추

```
collectors/  →  cache/  →  macro/ · stock/  →  chat/  →  api/  →  (프론트가 조회)
(외부 IO)      (게이트)     (결정적 엔진)        (LLM 계층)  (경계)
infra/ · auth/ · watchlist/ · rag/ 는 위 계층을 가로지르는 지원 모듈
```

**불변 규칙 (위반 = 구조 결함):**
- **판정·수치는 코드**(`macro/`·`stock/`): 국면 판정·정량 요약·기술적 지표는 **결정적 순수 함수**(dict→dict, LLM 미개입). 여기에 `openai`/`anthropic` import가 보이면 즉시 지적.
- **LLM은 설명만**(`chat/`): 엔진 결과를 서술·요약·라우팅한다. 판정을 새로 하지 않는다. 위험 차단은 결정적 가드레일(코드)이 먼저.
- **프론트가 실데이터 조회**: 표시 툴은 "무엇을 띄울지"만 정하고 실데이터는 프론트가 직접 조회(환각 차단). LLM이 화면 숫자를 지어내지 않게 하는 구조.
- **경계는 api/**: FastAPI가 유일한 HTTP 경계(로컬은 Lambda 스탠드인). 라우트는 조립만 하고 로직은 하위 계층에.

점검법: `macro/`·`stock/`에서 LLM import를 grep, `chat/`에서 판정 상수 재정의를 grep, 라우트에 비즈니스 로직이 뭉친 곳을 찾는다.

## 3. SSOT / 3중 일관성 — 같은 숫자를 두 번 타이핑하지 마라

임계값·라벨·기간은 **한 곳(상수)에서만** 정의하고, 파생물은 그로부터 **생성**한다:

```
macro.engine THRESHOLDS/REGIME_PARAMS  ─┬─→ build_criteria_text()  (프롬프트 기준표)
stock.constants INDICATOR_CONFIG        ├─→ 번들 indicator_config   (프론트 차트/대순환)
GRAND_CYCLE_STAGES 등 라벨 SSOT          └─→ Pydantic 필드 범위·프론트 카탈로그
```

프롬프트에 임계 숫자를 하드코딩하거나, 프론트가 6단계 라벨을 복제하거나, 스키마에 상수를 다시 적으면 **SSOT 위반**이다. 발견 시 "상수 import → 파생 생성"으로 통합 리팩토링을 지시한다. 테스트(`str(VIX_PANIC) in build_criteria_text()` 류)로 3중 일관성을 잠근다.

## 4. 경계 계약(Contract) 설계 — 검증이 아니라 설계

모듈 간 인터페이스는 QA가 사후 검증하기 전에 **설계 원칙**으로 미리 못 박는다:

- **고정 키 + 미산출은 None** (키 삭제 아님): `build_stock_summary`·판정 dict·view_context는 항상 같은 키셋을 반환하고, 값이 없으면 None. 소비자가 `.get` 지옥에 빠지지 않게.
- **partial_failure 보존**: 수집기 하나가 죽어도 번들 전체를 죽이지 않고 실패 목록을 실어 보낸다. 소비자는 섹션별 fallback.
- **항상 200 graceful**: 조회·수집 라우트는 예외를 삼켜 카운트/에러 필드로 반환(라우트는 200). 크래시로 프론트를 깨지 않는다.
- **표시 툴 vs 콘텐츠 툴**: 표시 툴은 `{ok:True}`만, 콘텐츠 툴은 서버가 실행해 실텍스트 되먹임. 이 이분법이 흐려지면 챗 흐름이 꼬인다.

계약이 바뀌면 **생산자·소비자·QA 세 곳**에 동시에 전파(한 곳만 고치면 경계면 버그).

## 5. 캐시 3원칙 — 구조로 강제

① 현재가·일봉 **무캐시**(형성 중 값) ② 실패 응답 미저장(`partial_failure` 있으면 set 생략) ③ 프리웜은 P2. `cache/` 게이트(화이트리스트 prefix `macro:`/`stock:meta:`/`kis:token:` 밖은 `CachePolicyError`)가 이를 **코드로** 막는지 확인한다. "캐시하면 빨라진다"는 유혹에 원칙1을 깨지 않게 감시.

## 6. 안전 불변식 — 협상 불가

- **매매 주문 API 0**: `order`/`buy`/`sell`/`place_order` 계열 함수·엔드포인트를 만들지 않는다(조회만). grep으로 상시 확인.
- **출처 귀속 + 면책**: 외부 콘텐츠(리포트·영상)·애널리스트 요약은 "…에 따르면"으로 귀속하고 에이전트 판정으로 제시하지 않는다.
- **위험 2층 차단**: 결정적 키워드 하드블록 + ML→LLM 재분류. 이 순서(가드레일이 먼저)가 구조적으로 지켜지는지.
- 안전 위반은 발견 즉시 다른 모든 작업을 멈추고 바로잡는다.

## 7. 리팩토링 — 행동 보존이 전부

리팩토링은 **동작을 바꾸지 않고 구조만 바꾸는 것**이다. 순서:

1. **테스트 green 확인** — 없으면 characterization 테스트(현재 동작을 그대로 고정하는 테스트)를 먼저 요청한다. 안전망 없는 리팩토링은 금지.
2. **작은 단계로 분할** — 한 번에 하나의 구조 변경. 각 단계 후 전체 테스트 green 유지.
3. **기능 변경과 분리** — 리팩토링 커밋과 기능 커밋을 섞지 않는다(리뷰·롤백 가능성).
4. **완료 후 QA 회귀** — 계약이 안 깨졌는지 qa-inspector에게 재검증 요청.

**리팩토링 트리거(냄새):** 거대 함수(단일 책임 위반)·중복 로직(SSOT 위반)·죽은 코드(미마운트/미참조)·순환 의존·전역 상태 결합·계층 누수·숨은 암묵 계약. 각각 진단(왜 문제)→방향(무엇을 어떻게)→담당→QA 순.

## 8. 기술부채 카탈로그

임시방편은 굳기 전에 기록한다. `_workspace/{week}_backend-architect_review.md`에:

```
## 기술부채 카탈로그
| ID | 위치(파일:라인) | 부채 | 왜 문제 | 우선순위 | 재발? | 담당 |
|----|----------------|------|---------|:--------:|:-----:|------|
| TD-01 | chat/chat.py:198 | 툴 결과 되먹임 로직 중복 | 표시/콘텐츠 분기가 chat·chat_stream 이중화 | P1 | - | llm |
```

우선순위: P0(안전·계층 위반, 즉시) · P1(SSOT·계약·중복, 이번 주차) · P2(가독성·명명, 여유 시). **재발** 표시로 같은 부채가 반복되면 구조를 바꾼다(개별 수정 반복 금지).

## 9. QA와의 분업 (겹치지 않게)

| | qa-inspector | backend-architect |
|--|--------------|-------------------|
| 본다 | 동작·경계면 shape·안전 grep·테스트 통과 | 계층·결합도·계약 설계·리팩토링·부채 |
| 시점 | 모듈 완성 직후(반응) | 설계 시(선제) + QA 발견의 근인 판단(사후) |
| 산출 | 통과/실패/미검증 | 진단 + 리팩토링 티켓 + 부채 카탈로그 |

**루프**: QA 발견 → 아키텍트가 일회성/구조적 판단 → 구조적이면 리팩토링 티켓 → 담당 수정 → QA 회귀. 같은 QA 실패 2회↑ → 아키텍트 개입(재발 방지).

## 10. 산출물 형식 (`_workspace/{week}_backend-architect_review.md`)

```
# 백엔드 아키텍처 리뷰 — {주차/범위}
## 계층·경계 진단     (판정=코드/LLM=설명 누수·프론트 조회 여부)
## SSOT / 3중 일관성   (중복 상수·라벨 이중화)
## 경계 계약           (고정키·partial_failure·graceful — 위반/개선)
## 캐시·안전 불변식     (무캐시·주문API 0·출처귀속 — grep 결과)
## 리팩토링 티켓        (파일:라인 · 왜 · 방향 · 담당 · QA 재검증 필요)

```

## 테스트 시나리오

**정상 흐름:** 리더가 "챗 계층 리팩토링 점검" 지시 → 아키텍트가 `chat/` 계층을 훑어 `chat()`·`chat_stream()`의 툴 되먹임 로직 중복(SSOT 위반)을 진단 → 공통 헬퍼 추출 리팩토링 티켓 발행(행동 보존, characterization 테스트 선행) → llm-engineer에 배정 → 완료 후 qa-inspector에 회귀 요청 → 부채 카탈로그에 TD로 기록.

**에러 흐름:** QA가 "번들에 없는 필드 접근" 경계면 버그를 2회 연속 보고 → 아키텍트가 근인을 "번들 계약이 문서화·타입화되지 않음(암묵 계약)"으로 진단 → 계약을 고정 키 + None + 테스트로 못 박는 구조 리팩토링 지시 → 재발 방지 후 QA 회귀 통과.
