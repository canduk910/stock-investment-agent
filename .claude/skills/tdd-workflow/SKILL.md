# TDD 워크플로우 — 테스트가 곧 스펙이다

이 프로젝트가 TDD에 특히 적합한 이유: 핵심 로직이 **임계값·경계값 중심의 규칙 엔진**이라 스펙(`invest_develop_PLAN.md`)에서 테스트를 기계적으로 도출할 수 있고, 안전 요건(주문 금지·단정 표현 배제)과 3중 일관성도 테스트로 고정할 수 있다. 테스트를 먼저 쓰면 "스펙을 코드로 번역"하는 단계가 명시적이 되어 경계값 해석 차이(예: vix=35는 발동인가 아닌가)가 구현 전에 드러난다.

## 사이클 (작업 단위마다 반복)

1. **Red** — 스펙 조항에서 실패하는 테스트를 먼저 작성하고, **실행해서 실패를 확인한다** (import 에러가 아니라 assertion 실패여야 한다 — 스켈레톤 함수는 만들되 본 구현은 하지 않는다). 실패 확인을 건너뛰면 "항상 통과하는 가짜 테스트"를 걸러낼 수 없다.
2. **Green** — 테스트를 통과시키는 최소 구현. 테스트에 없는 기능을 미리 만들지 않는다.
3. **Refactor** — 테스트를 초록으로 유지한 채 정리. 리팩토링 중 테스트를 고쳐야 한다면 그것은 리팩토링이 아니라 동작 변경이다 — 멈추고 스펙을 재확인한다.

**절대 규칙**: 테스트가 실패하면 구현을 고친다. 테스트나 임계값 상수를 구현에 맞춰 바꾸지 않는다. 상수를 바꿔야 할 정당한 이유가 있다면 그것은 스펙 변경이므로 리더에게 보고 후 진행한다.

## 계층별 테스트 전략

### 정량 엔진 (quant-engineer) — TDD 최적 지점

- 순수 함수(dict→dict)라 fixture·모킹 불필요. `quant-engine-rules` 스킬 §5의 경계값 케이스 목록이 곧 Red 단계의 테스트 목록이다 — 구현 전에 그 목록 전체를 테스트 파일로 옮겨라.
- 테스트 이름에 스펙 근거를 남겨라: `test_vix_35_does_not_trigger_override__plan_4_4` — QA와 후임자가 테스트↔스펙을 추적할 수 있다.

### 데이터 계층 (data-engineer) — 모킹 경계를 명확히

- **실 외부 API(KIS/FRED/DART/CNN)를 호출하는 단위 테스트 금지.** 느리고, 비결정적이고, 호출 한도를 태운다. HTTP 계층을 `responses`/`respx`/`unittest.mock`으로 대체하고, **기록된 실제 응답 JSON을 fixture로** 저장해 어댑터의 파싱·정규화 계약을 테스트한다.
- 캐시 정책 3원칙을 테스트로 고정한다:
  - 현재가 조회 경로에서 캐시 `set`이 호출되지 않음 (mock 캐시로 검증)
  - `partial_failure`가 비어있지 않은 응답은 캐시 저장이 생략됨
- 번들 API: fetch 하나에 예외를 주입 → 해당 섹션 `null` + `partial_failure` 기록 + 나머지 섹션 정상을 검증한다.
- 실 API 연동 확인용 테스트는 별도 마커(`@pytest.mark.live`)로 분리해 기본 실행에서 제외한다 — 키 없는 환경(CI·QA)에서도 전체 스위트가 돌아야 한다.

### LLM 계층 (llm-engineer) — 결정적 부분만 테스트

LLM 출력 자체는 비결정적이라 단위 테스트 대상이 아니다. 그 주변의 결정적 코드가 대상이다:

- **`build_criteria_text()` 출력에 상수 값이 포함되는지** — `assert str(VIX_PANIC) in text` 식으로. 이것이 3중 일관성의 자동 회귀 테스트다: 상수를 바꿨는데 프롬프트에 하드코딩 숫자가 남아 있으면 여기서 잡힌다.
- 시스템 프롬프트에 필수 블록([판정 출처 고정], 면책, 단정 금지 지침)이 존재하는지
- `chat()`의 text/popups 분리 — OpenAI 클라이언트를 mock하고 tool_calls 유무 케이스별 반환 shape 검증
- Pydantic 스키마 — `리스크요인=[]`이면 ValidationError, `종합의견="매수"`면 ValidationError (안전 요건이 타입에서 강제되는지)
- 인텐트 라우팅 — 6개 라벨 각각이 올바른 처리 경로로 분기하는지 (분류기 자체는 mock)

### 프론트엔드 (frontend-engineer) — 계약 지점 위주로 실용적으로

- 팝업 라우팅: popups[i].name 3종 각각이 대응 컴포넌트를 여는지
- `partial_failure` 렌더링: 실패 섹션은 "일시 조회 불가", 나머지는 정상 표시되는지
- 5주 과제이므로 스냅샷 남발·픽셀 검증은 하지 않는다. 경계면(계약) 분기만 테스트하면 충분하다.

## 모킹 함정

- **모킹 과다 주의**: 모든 것을 mock하면 "mock이 mock을 검증"하는 무의미한 테스트가 된다. 원칙 — 경계(HTTP, OpenAI, 캐시 저장소)만 mock하고, 그 안쪽 로직(파싱·계산·조립)은 실제 코드를 통과시킨다.
- mock의 반환 shape은 반드시 기록된 실제 응답(fixture)에서 가져온다. 손으로 지어낸 mock 응답은 경계면 버그를 숨긴다.

## Definition of Done (작업 완료 기준)

1. 테스트가 구현보다 먼저 작성·실행됐다 (Red 확인 기록)
2. 전체 테스트 스위트 통과 (해당 모듈만이 아니라 전체 — 회귀 확인)
3. 정상 경로 + 실패/경계 경로 테스트가 모두 존재한다
4. `_workspace/{week}_{agent}_{artifact}.md` 명세에 **"테스트 목록(스펙 근거 포함) → 구현" 순서로 기록**한다 — qa-inspector가 test-first 증거로 확인하는 자료다

## QA와의 관계

qa-inspector는 (1) 전체 스위트를 직접 실행하고 (2) `_workspace/` 명세의 테스트 목록이 스펙 경계 케이스를 커버하는지 (3) 구현에 맞춰 테스트가 변형된 흔적이 없는지 검증한다. 테스트가 없는 모듈은 QA에서 자동 실패다.
