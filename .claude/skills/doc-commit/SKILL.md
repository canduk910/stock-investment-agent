# doc-commit — 문서 정리 → 커밋 → 푸시

작업 구간이 끝날 때 "기억할 것을 문서로 남기고, 깔끔히 커밋하고, 올린다"를 한 번에 처리한다. 세 단계를 **순서대로** 수행하고, 각 단계 결과를 사용자에게 짧게 보고한다.

## Phase 1 — 진행 상황을 CLAUDE.md에 정리

### 무엇을 남기는가 (선별이 핵심)
`git status`·`git diff`(마지막 커밋 이후, 커밋이 없으면 전체)와 대화 맥락을 보고, **꼭 기억해야 할 것만** 추린다.

기준: **코드나 git 로그에서 자명하게 드러나지 않는 것.** 즉 설계 결정, 비자명한 제약·함정(gotcha), 모듈 간 계약(반환 shape·엔드포인트·필드명), 컨벤션, "왜 이렇게 했는가". 파일 목록·디렉토리 구조·구현 방법 자체는 넣지 않는다 — 코드를 읽으면 알 수 있어 중복이고 금방 낡는다.

한 줄 테스트: "이걸 안 적어두면 다음에 이 코드를 만질 때 헷갈리거나 실수할까?" → 예면 남기고, 아니면 버린다.

### 어디에 남기는가 (범위로 분류)
- **특정 기능/모듈에 국한** → 그 디렉토리 하위 `CLAUDE.md`. 예: `collectors/CLAUDE.md`, `frontend/CLAUDE.md`, `cache/CLAUDE.md`, `api/CLAUDE.md`. 이유: Claude Code는 해당 디렉토리에서 작업할 때 그 CLAUDE.md를 자동으로 컨텍스트에 올린다. 모듈 지식이 필요할 때만 로드되므로, 메인 CLAUDE.md가 비대해지지 않고 지식이 관련 작업 근처에 놓인다.
- **프로젝트 전반(교차 관심사)** → 메인 `CLAUDE.md`. 안전 원칙, 전 계층 공통 컨벤션, 하네스 트리거 등. 메인은 **lean하게** 유지 — 상세는 nested에 두고, 메인에는 핵심 + "문서 지도"(nested CLAUDE.md 위치 인덱스)만 둬서 프로젝트 전반 관점에서 어디를 봐야 하는지 참고할 수 있게 한다.

### 작성 규칙
- nested CLAUDE.md는 그 디렉토리 관점에서 **짧게**. 이미 있으면 갱신하되 **낡거나 중복된 항목은 지운다**(추가만 하지 않는다). 없으면 새로 만든다.
- 메인 CLAUDE.md에 프로젝트 전반 항목을 반영하고, nested 문서가 새로 생겼으면 문서 지도(인덱스)에 한 줄 추가한다.
- 변경 이력 테이블이 있는 프로젝트(예: 하네스)는 그 규약을 따라 한 줄 기록한다.

## Phase 2 — 간결한 커밋

1. **스테이징**: 이번 작업과 관련된 변경만 스테이징한다. 스테이징 전 반드시 확인 — `.env`·`.cache/`·토큰 파일·`node_modules` 등 시크릿/생성물이 포함되지 않는가(gitignore 준수). 실수로 추적되고 있으면 커밋을 멈추고 사용자에게 알린다.
2. **메시지는 간결하게**: 제목 한 줄(무엇을·왜, 명령형, 대략 50자 이내) + 필요 시 본문에 핵심 3~5개 bullet만. 변경 파일을 나열하지 않는다(diff가 이미 말한다). "무엇을 바꿨나"보다 "왜/무엇이 달라지는가"를 담는다.
3. **트레일러**: 커밋 메시지 끝에 이 세션의 표준 커밋 트레일러(`Co-Authored-By` 및 `Claude-Session`)를 하네스 규약대로 붙인다.
4. 커밋 직전, 스테이징 diff와 메시지를 한 번 눈으로 재확인한 뒤 커밋한다.

> 브랜치 주의: 팀이 PR 기반 흐름이면 이 스킬을 부르기 전에 피처 브랜치에 있어야 한다. 이 스킬은 **현재 브랜치**에 커밋하고 그 브랜치를 푸시하며, 사용자 의도(커밋→푸시)를 존중해 임의로 브랜치를 새로 만들지 않는다.

## Phase 3 — 푸시

1. 원격(remote) 존재를 확인한다(`git remote -v`). 없으면 푸시를 **멈추고** 사용자에게 알린다 — 원격 생성/연결은 사용자 몫이다(이 스킬이 임의로 원격을 만들지 않는다).
2. 현재 브랜치에 upstream이 없으면 `git push -u origin <branch>`로 설정하며 푸시하고, 있으면 `git push`.
3. **강제 푸시·히스토리 재작성 금지.**
4. 결과를 보고한다: 브랜치, 커밋 해시(짧게), 원격, 그리고 이번에 남긴 문서 파일 목록.

## 안전·경계

- 시크릿을 절대 커밋하지 않는다. 스테이징 diff에 키·토큰 문자열이 보이면 멈춘다.
- 푸시는 되돌리기 어렵다. 이 스킬은 사용자가 명시적으로 호출했을 때만 실행되며(호출 = 커밋·푸시 승인), 실행 직후 "무엇을 어느 브랜치→원격으로 올렸는지"를 명확히 보고한다.
- 세 단계 중 하나라도 막히면(예: 원격 없음, 시크릿 감지, diff 없음) 그 지점에서 멈추고 상황을 보고한다. 절반만 처리하고 조용히 넘어가지 않는다.

## 테스트 시나리오

**정상 흐름**: 여러 파일 변경 상태에서 `/doc-commit` → 기능별 변경은 각 디렉토리 CLAUDE.md에, 전반 사항은 메인 CLAUDE.md에 반영 → 시크릿 없음 확인 후 간결한 커밋 → 원격에 푸시 → 브랜치·해시·문서 목록 보고.

**에러 흐름(원격 없음)**: 원격 미설정 상태에서 호출 → Phase 1·2(문서화·커밋)까지 수행 → Phase 3에서 원격 부재 감지 → 커밋은 완료됐음을 알리고 "원격을 연결하면 푸시하겠다"고 보고 후 멈춤.
