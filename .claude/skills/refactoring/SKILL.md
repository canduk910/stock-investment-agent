# Refactoring — 행동 보존 리팩토링 규율

리팩토링은 **외부 동작을 바꾸지 않고 내부 구조만 개선**하는 것이다. 위험은 "구조를 고치다 조용히 동작을 바꾸는 것". 이 스킬은 그 위험을 규율로 막는다. `backend-architecture`가 *무엇을* 리팩토링해야 하는지(구조 진단)를 정한다면, 여기서는 *어떻게* 안전하게 하는지를 담는다.

## 1. 단 하나의 황금률 — 행동 보존

리팩토링 전후로 **관측 가능한 동작이 동일**해야 한다: 함수 입출력, API 응답 계약(shape·키·상태코드), 테스트 결과, 안전 불변식. 동작을 바꾸고 싶으면 그건 리팩토링이 아니라 기능 변경/버그 수정이다 — **별개 작업으로 분리**한다.

## 2. 전제 — 안전망(테스트)이 먼저

- **테스트 green 확인 후 시작.** 리팩토링은 tdd-workflow의 Red→Green→**Refactor** 중 마지막 단계다. green 위에서만 구조를 만진다.
- **커버리지가 없으면 characterization 테스트를 먼저 쓴다.** 현재 동작을 "있는 그대로"(설령 이상해 보여도) 고정하는 테스트다. 리팩토링은 그 동작을 보존하고, 잘못된 동작의 수정은 **그 다음** 별도 단계에서 한다. 안전망 없는 리팩토링은 금지.

## 3. 규율 — 작게, 분리해서

1. **작은 단계.** 한 번에 하나의 구조 변경. 각 단계 후 **전체 테스트를 돌려 green 확인**. red면 즉시 직전 상태로 롤백하고 더 작게 쪼갠다.
2. **기능 변경과 섞지 않는다.** 리팩토링 커밋과 기능/버그 커밋을 분리한다 — 리뷰·bisect·롤백이 가능하도록. "리팩토링하면서 김에 기능도" = 금지.
3. **한 번에 하나의 관심사.** 이름 바꾸기 / 함수 추출 / 이동을 한 커밋에 뒤섞지 않는다.
4. **Rule of Three (성급한 공통화 금지).** 중복이 3번째 나타나기 전엔 억지로 추상화하지 않는다. 추상화를 위한 추상화(YAGNI)는 오히려 부채다.

## 4. 냄새 → 리팩토링 카탈로그

| 냄새(smell) | 리팩토링 | 주의 |
|-------------|----------|------|
| 거대 함수(단일 책임 위반) | Extract Function | 추출 단위는 "한 문장으로 이름 붙는" 응집 |
| 중복 로직(SSOT 위반) | Extract Shared / 상수 단일화 | 우연한 중복과 진짜 중복 구분 |
| 죽은 코드(미참조·미마운트) | Delete | **참조 0을 grep으로 확인 후** 삭제(레거시 죽은 코드도) |
| 순환 의존 | 의존성 역전 / 지연 import | 지연 import는 로드 결합 차단용 — 남용 아닌지 |
| 긴 파라미터 목록 | Parameter Object | 함께 다니는 값만 묶는다 |
| 산탄총 수술(한 변경이 여러 곳) | Move/Merge(응집화) | 흩어진 책임을 한곳으로 |
| Feature Envy(남 데이터에 집착) | Move Function | 데이터 곁으로 로직을 옮긴다 |
| 조건 분기 중복 | Dispatch/전략화 | 과용 금지(단순 if가 나을 때 많다) |
| 원시값 강박 | Value Object 도입 | 신중 — 오버엔지니어링 경계 |

카탈로그는 도구지 목표가 아니다. "더 읽기 쉽고 바꾸기 쉬운가"가 기준이지, 패턴 적용 자체가 목적이 아니다.

## 5. 리팩토링 중에도 절대 안 바뀌는 것 (이 프로젝트)

이 불변식들은 "외부 동작"이므로 **보존 대상**이다. 리팩토링이 이 중 하나라도 흔들면 그건 리팩토링이 아니다:

- **매매 주문 API 0**(조회만) · **현재가·일봉 무캐시**(원칙1) · **실패 응답 미저장**
- **판정=코드 / LLM=설명** 계층 경계 · **SSOT(3중 일관성)** — 상수를 두 곳으로 복제하며 "정리"하지 않는다
- **경계 계약**: 고정 키 + 미산출 None · `partial_failure` 보존 · 조회/수집 **graceful 200**
- **출처 귀속 + 면책** · 위험 2층 가드레일 순서

## 6. 검증 체크리스트 (리팩토링 전/후 동일해야)

1. 전체 테스트 green — 개수·통과가 **동일**(리팩토링이 테스트를 깨거나 새 실패를 만들면 안 됨)
2. 계약 diff 0 — 경계면 shape(번들·툴·판정·view_context)이 불변
3. 안전 grep 불변 — 주문 API·hex 하드코딩·방향색 오용·현재가 캐시 0 유지
4. 커밋 분리 — 리팩토링 단독 커밋(기능 변경 없음)
5. (백엔드) 완료 후 **qa-inspector 회귀 검증** 요청

## 7. 멈춤 신호

- 테스트가 red → **즉시 롤백**, 더 작은 단계로 재시도(디버깅으로 밀어붙이지 않는다)
- 리팩토링이 **기능 변경을 강제** → 멈추고 별도 티켓으로 분리
- 테스트가 아예 없어 특성 파악 불가 → characterization 테스트부터(리팩토링 보류)

## 8. 리팩토링 티켓 형식

```
## RF-xx {한 줄 요약}
- 냄새: {거대 함수 / 중복 / 죽은 코드 / 순환 의존 …}
- 위치: {파일:라인}
- 리팩토링: {Extract Function / SSOT 통합 / Delete …}
- 행동 보존 근거: {기존 테스트 green / 신규 characterization 테스트}
- 담당: {에이전트} · QA 회귀: {필요/불요}
```

## 테스트 시나리오

**정상 흐름:** `chat.py`의 `chat()`·`chat_stream()`이 툴 결과 되먹임 로직을 중복(SSOT 위반)으로 갖고 있음 → 두 경로가 이미 테스트로 덮여 있는지 확인(green) → 공통 헬퍼로 Extract Function(한 단계) → 전체 테스트 green 재확인 → 계약(응답 shape) diff 0 확인 → 리팩토링 단독 커밋 → qa-inspector 회귀 요청.

**에러 흐름:** 죽은 코드로 보이는 `StockReport.jsx`를 삭제하려다 → grep으로 참조를 확인하니 특정 경로에서 아직 import 중 → 삭제 보류, "미마운트 여부"를 특성 테스트로 확정한 뒤에만 삭제(성급한 delete 금지). 또는: 함수 추출 후 테스트 1건 red → 즉시 롤백하고 추출 경계를 더 작게 다시 잡는다.
