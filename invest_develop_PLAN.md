## 0. 한 줄 요약

개인 투자자를 위한 금융 분석 어시스턴트. 거시 시장 국면(매크로)을 규칙 기반으로 판정해 권장 현금비중을 산출하고, 보유·관심 종목을 기본적/기술적으로 분석하며, 챗봇이 이 모든 정보를 종합해 자연어로 상담한다. 대화 중 필요할 때 매크로/종목 화면을 실시간 팝업으로 띄운다. **매매 주문은 실행하지 않고 제안까지만 한다.**

---

## 1. 핵심 설계 원칙 (변경 금지)

1. **판단 보조자, 자동 매매자 아님** — 매수/매도 주문은 절대 실행하지 않는다. 모든 출력은 "제안"까지만. (책임 소재 리스크 회피)
2. **환각 차단** — 시세·재무 등 모든 구체적 숫자는 조회한 실제 데이터에서만 인용. LLM이 숫자를 지어내지 않는다.
3. **규칙과 LLM의 역할 분리** — 매크로 국면 판정은 규칙 기반 코드가 수행. LLM은 그 결과를 "설명"만 한다(재판정 금지).
4. **[v1.1 확장] 3중 일관성(Triple Consistency)** — 판정 임계값은 다음 세 곳에서 반드시 같은 값을 가리켜야 한다:
   - ① **코드 상수** (`THRESHOLDS` — 판정의 원본)
   - ② **LLM 프롬프트 기준표 텍스트** (①에서 자동 생성)
   - ③ **LLM 응답 스키마의 필드 범위** (Pydantic 검증 — §6.5 참조)
   > 비유: 저울(코드), 저울 눈금 설명서(프롬프트), 검수 기준표(스키마)가 서로 다른 단위를 쓰면 아무리 정확히 재도 보고서가 틀어진다. 세 문서를 한 원본에서 복사하는 것.
   > 레퍼런스 리포는 이를 "3중 일관성 필수 — 변경 시 3곳 동시 수정"으로 명문화하고 있다. 우리는 ①→② 자동 생성으로 이미 절반을 해결했으므로, ③(응답 스키마)만 추가하면 된다.
5. **컴포넌트/엔진 재사용** — 보유종목 분석 엔진을 관심종목에도 그대로 재사용. 팝업 UI는 독립 페이지로도 쓴다.

## 2. 기술 스택

- **LLM API**: OpenAI (gpt-4o), function calling 사용
- **증권 데이터**: KIS(한국투자증권) Open API — 시세, 지수, 계좌 잔고, 관심종목 시세, 종목정보
- **재무·공시**: DART API
- **거시 지표**: FRED API(금리차/신용스프레드/달러지수/GDP), CNN 공포탐욕지수(비공식 스크래핑), VIX(야후파이낸스 또는 FRED)

- **[v1.1 추가] 스키마 검증**: Pydantic — LLM의 구조화 응답(JSON) 검증용

## 3. 3대 모듈 구조

데이터가 한 방향으로 흐르는 파이프라인이다: 매크로 → 포트폴리오 → 후보종목, 그리고 LLM 챗봇이 전 정보를 종합한다.

### 모듈 1 — 매크로 분석 (시장 국면 판정)
- 7개 지표를 규칙 기반으로 점수화 → 4단계 국면 판정: **회복 → 확장 → 과열 → 수축**
- 국면별 권장 현금비중: 회복 80% / 확장 60% / 과열 40% / 수축 20% (방어적일수록 현금↑)
- **주의**: 현금비중 매핑은 "시장대응" 관점. 수축장에서 현금을 가장 많이(방어), 회복장에서 가장 적게(공격) 가져간다.
- **[v1.1]** 판정 결과는 현금비중 외에 "실행 파라미터 세트"(§4.5)까지 산출해 모듈 2·3의 종목 판단 기준으로 주입한다.

### 모듈 2 — 포트폴리오 분석
- KIS 계좌 잔고에서 보유종목 자동 수집
- 종목별 리포트: 기본적 분석(DART 재무) + 기술적 분석(이동평균/RSI 등)
- **[v1.1]** 코드가 먼저 정량 요약(CAGR·밸류에이션 판정, §6.5)을 계산하고, LLM은 그 위에서 서술한다
- 국면 기반 비중 조정 가이드 (제안만)
- 챗봇 상담: 매크로 국면 + 종목 분석을 프롬프트로 조립해 LLM 호출

### 모듈 3 — 신규 후보종목 관리
- 모듈 2의 분석 엔진을 관심종목(워치리스트)에 그대로 재사용
- 차이: 질문 방향이 "비중 조정"이 아닌 "신규 진입(살까/언제)"
- 워치리스트 CRUD + 진입 신호 판정(국면과 결합) + (선택)목표가 알림
- **5주 내 시간 부족 시 가장 먼저 축소 가능한 모듈** (분석 엔진은 모듈 2에서 이미 완성)

## 4. 매크로 7개 지표 — 데이터 소스 및 임계값

| 지표 | 소스 | 키 필요 | 수축 신호 | 과열 신호 | 그룹/가중치 |
|---|---|---|---|---|---|
| 장단기 금리차 (T10Y2Y) | FRED | 무료 키 | < 0 (역전) | 0~0.5 | 신용·금리 / 2 |
| 하이일드 신용스프레드 (BAMLH0A0HYM2) | FRED | 무료 키 | > 5.0 | < 3.0 | 신용·금리 / 2 |
| VIX | 야후파이낸스/FRED | 무료 | > 28 | < 14 | 변동성·심리 / 1 |
| 공포탐욕지수 | CNN(비공식) | 스크래핑 | < 25 | > 75 | 변동성·심리 / 1 |
| 달러 강도 (DTWEXBGS) | FRED | 무료 키 | 급등 | 약세 | 시장구조 / 1 |
| 버핏지수 | KRX 시총 ÷ GDP(FRED) | 무료 키+계산 | — | > 150 | 밸류 / 1 |
| 섹터 집중도 | KIS 업종지수로 계산 | KIS 키 | 쏠림 심화 | — | 시장구조 / 1 |

**중복 보정**: 공포탐욕지수 자체가 VIX·신용스프레드 등 7요소의 종합이라 정보가 겹친다. 신용·금리 그룹은 가중치 2, 변동성·심리 그룹은 가중치 1로 차등 투표시켜 중복 과대반영을 막는다.

**구현 순서 권고**: WEEK 07에는 3~4개 지표(금리차·신용스프레드·VIX·공포탐욕)로 파이프라인을 먼저 완성하고, 나머지는 확장으로 둔다.

**주의**: CNN 공포탐욕지수는 공식 API가 없어 스크래핑/파이썬 래퍼(`fear-and-greed` 등)에 의존. CNN 페이지 구조 변경 시 깨질 수 있으니 실패 처리 필수.

### 4.1 [v1.1 · P1] 투표 방식 개선 — 정수 투표 → 분수 점수제

레퍼런스 리포의 `macro_cycle.py`는 지표 하나가 국면 하나에만 "몰표"하지 않고, **하나의 지표가 여러 국면에 점수를 나눠 배분**한다(합계 1.0의 분수 점수). 예: 금리차 0~1.5 구간이면 확장 0.5 + 과열 0.5.

- **왜 좋은가**: 정수 몰표는 경계값 근처에서 판정이 뚝뚝 끊긴다(금리차 0.51 → 확장 몰표, 0.49 → 과열 몰표). 분수 배분은 "애매한 상태"를 표현할 수 있어 판정이 부드럽다.
  > 비유: 선거에서 유권자가 1표를 한 후보에게만 주는 것(정수 투표) vs 선호도에 따라 0.7표·0.3표로 나누는 것(분수 투표). 후자가 "박빙 상황"을 훨씬 정확히 반영한다.
- **적용 방식**: 기존 `THRESHOLDS` 경계 3구간(수축/정상/과열)은 유지하되, "정상" 구간을 인접 국면에 분할 배분한다. 그룹 가중치(신용·금리 ×2)는 그대로 곱한다.
- **범위 조절**: 1차 구현은 기존 정수 투표로 완성 → 2차 정교화 때 분수 점수제 전환 (WEEK 07 계획과 일치).

### 4.2 [v1.1 · P1] 신뢰도(confidence) 공식 개선

기존: `1위 득표 / 전체 득표` 비율. 개선: **`(1위 점수 − 2위 점수) × 배율`, 최대 100** (레퍼런스: `min(100, (top−second)×200)`).

- **왜 좋은가**: 비율 방식은 "1위 40%, 2위 39%"처럼 초박빙이어도 medium이 나올 수 있다. 1·2위 **격차** 방식은 "판정이 얼마나 확실히 갈렸는가"를 직접 측정한다.
  > 비유: 마라톤에서 우승자의 실력은 기록 자체보다 "2위와의 격차"가 더 잘 말해준다.

### 4.3 [v1.1 · P2] 하이스테리시스 — 국면 토글(깜빡임) 방지

하이스테리시스(hysteresis): 상태가 경계값 근처에서 왔다갔다할 때, **이전 상태를 기억해 경계에 ±버퍼를 두고 잘 안 바뀌게** 만드는 기법. 보일러 온도조절기가 딱 20.0°C에서 켜졌다 꺼졌다를 반복하지 않도록 19.5~20.5°C 여유 구간을 두는 것과 같은 원리다.

- 레퍼런스 구현: 공포탐욕 경계 ±5점, 버핏지수 경계 ±0.05 버퍼. 직전 국면(`previous_regime`)을 저장해두고, 새 판정이 경계 버퍼 안이면 이전 국면을 유지한다.
- **우리 적용**: 매크로 판정 결과를 DynamoDB에 일 단위로 저장(§7)하고, 판정 시 직전 국면을 조회해 각 지표 임계값에 ±버퍼를 적용. "어제는 과열, 오늘은 확장, 내일 다시 과열" 같은 신뢰를 깎는 깜빡임을 막는다.
- **단, 예외**: 아래 오버라이드(4.4)는 하이스테리시스를 무시하고 즉시 발동 — 시장 패닉엔 즉각 반응해야 하기 때문.

### 4.4 [v1.1 · P2] 강제 오버라이드 규칙 — 극단 상황 즉시 반응

가중 투표는 평균을 내기 때문에 극단 신호가 희석될 수 있다. 레퍼런스는 특정 조건에서 투표를 건너뛰고 국면을 강제한다:

| 조건 | 강제 결과 | 근거 |
|---|---|---|
| VIX > 35 | 수축 (즉시, 버퍼 무시) | 시장 패닉 — 다른 지표가 뭐라 하든 방어 |
| HY 스프레드 > 10% | (역발상) 극단 공포 = 축적 기회로 별도 표기 [P2 선택] | 하워드 막스식 "시계추 극단" 관점 |

- **우리 적용 최소안 [P1]**: `VIX > 35 → 무조건 수축` 한 줄만 넣는다. 구현 비용이 거의 없고 안전성이 크게 오른다.
- 역발상 오버라이드는 "극단 공포 = 매수 기회" 논리라 초심자에게 위험 소지가 있으므로, 넣는다면 판정을 바꾸지 말고 **설명 텍스트에 참고로만** 언급한다 (원칙 1·2와 충돌 방지).

### 4.5 [v1.1 · P1] 국면 → 실행 파라미터 세트 확장

기존엔 국면이 **현금비중 하나**만 산출했다. 레퍼런스(`macro_regime.py`의 `REGIME_PARAMS`)는 국면마다 **종목 판단에 쓸 구체 기준 세트**를 함께 정의한다. 이를 차용하면 모듈 1(매크로)과 모듈 2·3(종목)의 연결이 "분위기 전달"에서 "수치 기준 전달"로 격상된다.

```python
# 국면별 실행 파라미터 — 매크로가 종목 판단의 "눈높이"를 정한다
REGIME_PARAMS = {
    #          현금비중  종목당한도  PER상한  PBR상한  # 신중할수록 눈높이가 깐깐해짐
    "회복":  {"cash": 20, "single_cap": 5, "per_max": 20, "pbr_max": 2.0},
    "확장":  {"cash": 40, "single_cap": 4, "per_max": 15, "pbr_max": 1.5},
    "과열":  {"cash": 60, "single_cap": 3, "per_max": 12, "pbr_max": 1.2},
    "수축":  {"cash": 80, "single_cap": 0, "per_max": None, "pbr_max": None},  # 신규 진입 자제
}
```

> ※ 위 표의 현금비중은 예시 스케일이며, 기존 §3의 "회복 80(주식)/… " 표기와 관점(주식비중 vs 현금비중)을 통일해 한 곳에서만 정의할 것.
> ※ `single_cap` = 종목당 최대 비중(%). 수축기 0은 "신규 진입 제안 자체를 하지 않음"의 의미.

- **활용처**:
  - 모듈 2: "국면상 PER 상한이 12인데 이 종목은 18 → 비중 축소 검토 제안" 식의 **근거 있는 제안** 생성
  - 모듈 3: 진입 신호 판정에 `single_cap`·`per_max`를 게이트로 사용 (수축기엔 진입 신호 미발생)
  - 프롬프트: 파라미터를 시스템 프롬프트에 주입 → LLM 설명이 항상 국면 기준을 인용
- 이 표 역시 원칙 4(3중 일관성)에 따라 코드 상수 1곳에 정의하고 프롬프트 텍스트를 자동 생성한다.

### 4.6 [v1.1 · P2] 레벨 + 방향(모멘텀) 신호

기존 임계값은 전부 "현재 레벨"만 본다. 레퍼런스는 레벨과 함께 **방향**(금리차 확대 중/축소 중, 신용스프레드 확대 중/축소 중)을 별도 신호로 넣고 소폭 가점한다.

- 예: 신용스프레드가 6개월 전 대비 +50% 이상 확대 → 수축 가점, −30% 이상 축소 → 회복 가점
- **왜 좋은가**: 같은 VIX 25라도 "20에서 오르는 중"과 "35에서 내려오는 중"은 정반대 국면이다. 레벨만 보면 이 둘을 구분 못 한다.
  > 비유: 체온 37.5°C라는 숫자만으로는 "열이 오르는 중"인지 "해열 중"인지 모른다. 방향이 진단을 바꾼다.
- **구현**: FRED에서 시계열을 어차피 받으므로, 최근 값과 N개월 전 값 비교 한 줄이면 된다. 다만 지표당 신호가 2배로 늘어 튜닝 부담이 있으니 P2.

## 5. 팝업(생성형 UI) 아키텍처

대화 중 LLM이 관련 화면을 실시간 팝업으로 띄운다. 작동 원리:

1. 사용자 질문 → OpenAI 호출 (function calling 도구 정의 포함)
2. LLM이 **답변 텍스트 + 툴 콜(어떤 팝업을 띄울지)**을 함께 반환
3. 백엔드가 `text`(말풍선용)와 `tool_calls`(팝업 지시)를 분리해 프론트로 전달
4. 프론트엔드: text는 말풍선에, 툴 콜은 해당 컴포넌트 팝업으로
5. **팝업의 실제 데이터(시세/재무 숫자)는 LLM이 아니라 프론트가 KIS/DART에서 직접 조회해 채움** (환각 차단 + 항상 최신)

핵심: **페이지는 미리 만들고, LLM은 "무엇을 언제 띄울지"만 결정.** LLM이 HTML을 통째로 생성하지 않는다.

### 팝업 툴 정의 (OpenAI function calling 형식)
- `show_macro_dashboard(highlight)` — highlight: regime|cash_ratio|indicators
- `show_stock_report(ticker, stock_name, focus)` — focus: fundamental|technical|both
- `show_watchlist(sort_by)` — sort_by: registered|change_rate|near_target

각 도구 description에 "언제 호출하는지"와 "언제 호출하지 않는지"를 모두 명시한다(오발동 방지). 파라미터는 enum으로 자유도를 묶는다.

### 5.1 [v1.1 · P1] 팝업 데이터 로딩 — 번들 API 패턴 + 부분 실패 보존

종목 리포트 팝업 하나가 필요로 하는 데이터는 최소 4종(시세, 재무, 밸류에이션, 차트)이다. 프론트가 이를 각각 호출하면 **N+1 문제**(화면 하나 그리는 데 API를 N번 연쇄 호출해 느려지는 패턴)가 생긴다. 레퍼런스는 이 문제로 대시보드 로딩이 10~20초까지 갔다가, 번들 API로 2~5초로 줄였다.

- **번들 엔드포인트 1개**: `GET /api/detail/{ticker}/bundle` → 백엔드(Lambda)가 KIS·DART를 **병렬**(ThreadPool)로 동시 조회해 `{basic, financials, valuation, summary, partial_failure}` 하나로 반환.
  > 비유: 식당에서 반찬을 하나씩 시켜 5번 왕복하는 대신, 백반 한 상으로 한 번에 받는 것.
- **부분 실패 보존(partial_failure)**: 4종 중 1종(예: DART)이 실패해도 에러로 전체를 죽이지 않는다. 성공한 섹션은 채우고, 실패 섹션은 `null` + `partial_failure: ["financials"]` 리스트에 기록 → 팝업은 "재무 데이터 일시 조회 불가" 안내와 함께 나머지를 정상 표시.
  > 비유: 백반에서 국이 떨어졌다고 상 전체를 물리지 않고, "국은 품절입니다" 쪽지와 함께 나머지를 내는 것.
- 이 패턴은 `show_macro_dashboard`(지표 7종 병렬 수집)에도 동일하게 적용한다. 특히 CNN 스크래핑 실패 시 graceful 처리(§10)가 자연스럽게 이 구조 안에서 해결된다.

### 6.1 규칙 기반 매크로 엔진 (`macro_engine.py`)

```python
# 임계값을 한 곳에 정의 — 코드와 LLM 설명이 같은 값을 공유
THRESHOLDS = {
    "yield_spread": {"수축": "< 0", "과열": "0 ~ 0.5", "정상": "> 0.5"},
    "hy_spread":    {"수축": "> 5.0", "과열": "< 3.0", "정상": "3.0 ~ 5.0"},
    "vix":          {"수축": "> 28", "과열": "< 14", "정상": "14 ~ 28"},
    "fear_greed":   {"수축": "< 25", "과열": "> 75", "정상": "25 ~ 75"},
    "buffett":      {"과열": "> 150", "정상": "<= 150"},
}
CASH_RATIO = {"회복": 80, "확장": 60, "과열": 40, "수축": 20}

# [v1.1] 극단 오버라이드 — 가중투표보다 우선 (하이스테리시스도 무시)
VIX_PANIC = 35  # VIX가 이 값을 넘으면 무조건 수축

def score_indicators(data: dict) -> dict:
    """각 지표가 어느 국면에 투표하는지 평가. 신용·금리=가중치2, 변동성·심리=가중치1."""
    votes = {"회복": 0, "확장": 0, "과열": 0, "수축": 0}
    drivers = []
    if data["yield_spread"] < 0:
        votes["수축"] += 2; drivers.append(("장단기 금리차 역전", "수축", 2))
    elif data["yield_spread"] > 0.5:
        votes["확장"] += 1
    if data["hy_spread"] > 5.0:
        votes["수축"] += 2; drivers.append(("신용스프레드 확대", "수축", 2))
    elif data["hy_spread"] < 3.0:
        votes["과열"] += 1
    if data["vix"] > 28:   votes["수축"] += 1
    elif data["vix"] < 14: votes["과열"] += 1
    if data["fear_greed"] > 75:
        votes["과열"] += 1; drivers.append(("극단적 탐욕 심리", "과열", 1))
    elif data["fear_greed"] < 25:
        votes["수축"] += 1; drivers.append(("극단적 공포 심리", "수축", 1))
    if data["buffett"] > 150: votes["과열"] += 1
    return {"votes": votes, "drivers": drivers}

def judge_regime(data: dict, previous_regime: str | None = None) -> dict:
    """국면 판정 + 설명용 재료(votes, key_drivers, raw_data) 반환.

    [v1.1] previous_regime을 받아 하이스테리시스 적용(P2),
           VIX 패닉 오버라이드는 최우선(P1).
    """
    # [v1.1 · P1] 오버라이드 — 투표 이전에 검사
    if data["vix"] > VIX_PANIC:
        return {
            "regime": "수축", "recommended_cash_ratio": CASH_RATIO["수축"],
            "confidence": "high", "votes": None,
            "key_drivers": [("VIX 패닉 임계 초과", "수축", 99)],
            "override": f"VIX {data['vix']} > {VIX_PANIC} — 강제 수축",
            "raw_data": data,
        }

    r = score_indicators(data)
    votes = r["votes"]
    regime = max(votes, key=votes.get)

    # [v1.1 · P2] 하이스테리시스: 1·2위 격차가 작고 2위가 직전 국면이면 유지
    ranked = sorted(votes.items(), key=lambda x: x[1], reverse=True)
    top, second = ranked[0], ranked[1]
    if previous_regime and second[0] == previous_regime and (top[1] - second[1]) <= 1:
        regime = previous_regime

    # [v1.1 · P1] 신뢰도: 1·2위 "격차" 기반 (비율 방식 대체)
    gap = top[1] - second[1]
    conf = "high" if gap >= 3 else "medium" if gap >= 2 else "low"

    return {
        "regime": regime,
        "recommended_cash_ratio": CASH_RATIO[regime],
        "confidence": conf, "votes": votes,
        "key_drivers": r["drivers"], "raw_data": data,
        "override": None,
    }
```

### 6.3 OpenAI 호출 (`chat.py`)

```python
from openai import OpenAI
client = OpenAI()  # OPENAI_API_KEY 환경변수

TOOLS = [ ... ]  # show_macro_dashboard / show_stock_report / show_watchlist (function calling 형식)

def chat(user_message, judgement, history):
    messages = [{"role":"system","content":build_system_prompt(judgement)}]
    messages.extend(history)
    messages.append({"role":"user","content":user_message})
    resp = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=TOOLS, tool_choice="auto")
    msg = resp.choices[0].message
    result = {"text": msg.content or "", "popups": []}
    if msg.tool_calls:
        for c in msg.tool_calls:
            result["popups"].append({"name": c.function.name, "args": c.function.arguments})
    return result  # text=말풍선, popups=팝업 트리거
```

- `judgement`는 매 호출마다 최신 국면을 시스템 프롬프트에 주입(챗봇이 항상 현재 국면 인지)
- 응답을 text(말풍선) / popups(팝업 지시)로 분리 반환

### 6.4 종목 분석 프롬프트 (모듈 2·3 공용)

`holding_status` 하나로 보유("비중 조정 관점")/관심("신규 진입 관점") 분기.

입력: PER/PBR/매출·영업이익 추이(DART), 현재가/이평선/RSI/52주위치(KIS), 시장 국면, **[v1.1] 코드 사전계산 정량 요약(§6.5)**.
지침: 단정적 매수/매도 금지, 제공 수치 외 정보 추가 금지.

### 6.5 [v1.1 · P1] 종목 종합리포트 — 정량 요약은 코드가, 서술은 LLM이 (신규)

레퍼런스의 종합리포트 구조에서 가장 배울 점: **LLM에게 "분석해줘"라고 통째로 맡기지 않고, 코드가 먼저 결정적(deterministic) 정량 요약을 계산해 프롬프트에 "사전 계산값"으로 주입**한다. LLM은 그 숫자들 위에서 서술만 한다. 이는 우리 원칙 2·3(환각 차단, 역할 분리)을 매크로뿐 아니라 **종목 분석에도** 동일하게 관철하는 것이다.

#### (a) 코드가 계산하는 정량 요약 (`stock_summary.py`)

```python
def build_stock_summary(basic, financials, valuation) -> dict:
    """DART 재무 + KIS 시세에서 결정적 요약 산출. LLM 미개입."""
    return {
        # 성장성: CAGR(연평균 성장률 — 여러 해의 성장을 매년 균일 성장으로 환산한 값)
        "rev_cagr": ...,   # 매출 CAGR (기간: 확보한 재무 연수)
        "op_cagr": ...,    # 영업이익 CAGR
        # 밸류에이션: 현재 PER/PBR을 "그 종목 자신의 과거 평균"과 비교
        "current_per": ..., "avg_per": ...,   # avg = 조회 기간 평균
        "per_vs_avg": ...,  # (현재-평균)/평균 %, 예: -18.2 → 평균보다 18% 저렴
        "valuation_label": ...,  # 평균 대비 -10% 미만=저평가 / +10% 초과=고평가 / 그 외=적정
        # 기술적: KIS 일봉에서 계산
        "rsi": ..., "ma20_gap_pct": ..., "pos_52w_pct": ...,  # 52주 밴드 내 위치(0~100%)
    }
```

- **핵심 아이디어 — 자기 자신과의 비교**: "PER 15는 싼가 비싼가"는 절대 기준이 없지만, "이 종목의 5년 평균 PER 대비 −18%"는 근거가 명확하다. 판정 라벨(저평가/적정/고평가)도 ±10% 임계값으로 **코드가** 붙인다. LLM은 이 라벨을 바꿀 수 없고 이유만 풀어 쓴다.
  > 비유: 건강검진에서 의사가 혈압을 재는(코드) 것과, 그 수치가 무슨 의미인지 설명해주는(LLM) 것의 분리. 설명하는 사람이 수치를 다시 재거나 고치면 안 된다.
- 임계값(±10% 등)은 §1의 3중 일관성 원칙에 따라 상수 1곳 정의 → 프롬프트 기준표 자동 생성.

#### (b) [P2] LLM 리포트의 구조화 출력 + 스키마 검증

챗봇의 일반 답변은 자유 텍스트로 두되, **"종목 리포트 생성" 경로만큼은** LLM이 정해진 JSON 구조로 응답하게 하고 Pydantic으로 검증한다. 레퍼런스는 이 방식으로 리포트를 DB에 누적 저장해 과거 평가와 비교까지 지원한다.

```python
from pydantic import BaseModel, Field
from typing import Literal

class StockReport(BaseModel):
    종합의견: Literal["긍정적", "중립", "신중"]   # "매수/매도" 단정 라벨은 원칙 1에 따라 배제
    요약: str
    투자포인트: list[str] = Field(max_length=3)
    리스크요인: list[str] = Field(min_length=1, max_length=3)  # 리스크 최소 1개 강제 — 장밋빛 일변도 방지
    국면정합성: str      # 현재 국면 기준에서 이 종목이 적절한지 (REGIME_PARAMS 인용)
    면책고지: str        # 필수 필드 — 누락 시 검증 실패
```

- **왜 좋은가**:
  1. `리스크요인 min_length=1`, `면책고지` 필수화처럼 **안전 요건을 스키마 레벨에서 강제**할 수 있다. 프롬프트 지시("리스크를 꼭 언급해")는 어길 수 있지만, 검증 실패 시 재시도시키면 구조적으로 보장된다. → §10 체크리스트의 상당 부분이 자동 검증으로 넘어간다.
  2. 리포트를 DynamoDB에 저장해 **리포트 히스토리**(같은 종목의 과거 평가 비교)가 공짜로 생긴다 — 시연 때 강력한 데모 포인트.
  3. `종합의견`을 enum(허용 값 목록으로 제한하는 타입)으로 묶어 "반드시 오른다" 류의 표현이 등급 필드에 침투하는 것을 원천 차단.
- 검증 실패 시: 1회 재요청 → 재실패 시 정량 요약(a)만으로 리포트 표시 + "AI 서술 생성 실패" 안내 (부분 실패 보존 철학, §5.1과 동일).

#### (c) 종합리포트 팝업/페이지 구성 (레퍼런스 DetailPage 차용, 소형화)

레퍼런스는 종목 상세를 탭 구조(재무 / 종합리포트[기본적·기술적·AI리포트·히스토리 서브탭])로 구성한다. 우리 `show_stock_report` 팝업도 동일한 3단 구성을 소형화해 적용:

1. **상단 카드**: (a)의 정량 요약 — CAGR 카드 + PER/PBR vs 평균 + 저평가/적정/고평가 배지 (숫자·판정 모두 코드 산출)
2. **중단**: `focus` 파라미터에 따라 기본적(재무 추이 테이블) 또는 기술적(이평/RSI) 섹션
3. **하단**: (b)의 LLM 서술 리포트 + 면책 고지 고정 노출

### 지표 캐시 (ElastiCache)
외부 API는 호출 한도가 있으므로 매크로 지표를 캐시. 키 예: `macro:T10Y2Y`, TTL은 지표 갱신 주기에 맞춤(일 단위 지표는 수 시간).

#### [v1.1 · P1] 캐시 정책 3원칙 (레퍼런스 운영 경험 차용)

레퍼런스는 캐시 관련 버그를 두 차례 크게 수정했고, 그 교훈이 커밋 이력에 명시돼 있다. 처음부터 반영한다:

1. **현재가는 캐시 금지** — 시세(현재가·등락률)는 팝업을 열 때마다 KIS에서 직접 조회. 캐시하면 "환각은 아닌데 낡은 숫자"를 보여주게 되고, 이는 사용자 신뢰 관점에서 환각과 다를 바 없다. **메타 정보(섹터·52주 고저·PER·PBR)만 수 시간 캐시** 허용.
   > 레퍼런스는 실제로 6개 캐시 레이어를 전부 우회하도록 고치는 대공사를 했다 — 처음부터 "가격은 캐시 대상이 아니다"로 설계하면 이 공사가 필요 없다.
2. **실패 응답은 캐시에 저장하지 않는다** — 부분 실패(§5.1)가 섞인 응답을 일일 캐시에 넣으면, 원인(예: API 키 미설정)이 해결돼도 자정까지 깨진 데이터가 계속 표시된다. 캐시 저장 전 `partial_failure` 검사 → 실패 포함 시 저장 생략, 다음 요청이 재시도.
3. **[P2] 프리웜(pre-warm)** — 매일 자정 직후(예: KST 00:05) 스케줄러(EventBridge → Lambda)가 매크로 지표를 미리 수집·캐시. 아침 첫 사용자가 콜드 스타트(캐시가 비어 외부 API를 전부 기다리는 상태) 없이 즉시 응답받는다. "매주 자동 실행" 계획을 일 단위 프리웜으로 구체화하는 것.

## 12. 인텐트 분류 (사용자 질문 라우팅)

사용자 질문을 6개 인텐트 중 하나로 자동 분류해, 적절한 모듈/팝업으로 라우팅하고 위험 질문을 차단한다. 보험 인텐트 데이터셋(coverage_check 등 6분류) 패턴을 차용하되, 투자 도메인에 맞게 재설계했다.

핵심: 안내 데스크 역할. 두서없는 질문도 맞는 기능으로 보낸다. `risk_guardrail`은 별도 보안 게이트로, 다른 인텐트와 절대 섞지 않는다.

### 6개 인텐트

| 인텐트 | 처리 경로 | 연결 팝업 툴 | 질문 예시 |
|---|---|---|---|
| `macro_view` | 규칙 엔진 결과 설명 | show_macro_dashboard | "시장 어때", "현금 비중 얼마나 가져가야 해", "왜 신중 단계야" |
| `stock_analysis` | 종목 리포트 생성 | show_stock_report | "삼성전자 어때", "PER 적정해", "차트상 매수 구간이야" |
| `portfolio_advice` | 보유현황+국면 비교 | (M2 분석 + 텍스트) | "내 포트폴리오 어때", "비중 어떻게 바꿔야 할까", "뭘 줄여야 해" |
| `watchlist_mgmt` | 워치리스트 CRUD | show_watchlist | "관심종목 보여줘", "관심목록에 추가해줘", "목표가 근접한 거 있어" |
| `general_qa` | 정적 설명 / RAG | (없음, 텍스트 응답) | "RSI가 뭐야", "분산투자가 왜 중요해", "복리 어떻게 계산해" |
| `risk_guardrail` | **차단·경고** | (차단 응답) | "무조건 오르는 종목 알려줘", "내부정보로 살 종목", "빚내서 투자할까" |

### risk_guardrail — 투자 도메인의 차단 대상

보험은 사기·개인정보가 위험이었다면, 투자는 위험 성격이 다르다. 반드시 차단·완화할 3유형:
1. **단정적 예측 요구** — "이거 반드시 오르지?" → 시스템 프롬프트 "단정 표현 금지"가 발동. 미래 수익 보장 시 책임 문제.
2. **미공개·내부 정보 유도** — "내부정보로 살 종목" → 위법 영역, 정보 제공 거절.
3. **과도한 위험 감수 조장** — "전 재산 몰빵해도 돼", "빚내서 투자할까" → 답 대신 위험 환기 + 분산 안내로 방향 전환.
4. (추가) **시세조종·부정거래** — "주가 띄우는 법" → 명백한 위법, 거절.

### 학습 데이터셋

`investment_intent_dataset_1000.txt` (이 문서와 함께 제공). 형식: `질문<TAB>인텐트라벨`, 총 1000개.

### 분류 구현 방향

- **방식**: OpenAI로 질문을 받아 6개 라벨 중 하나로 분류(few-shot 또는 파인튜닝). 초기엔 few-shot 프롬프트로 시작.
- **라우팅**: 분류 결과 → 해당 모듈/팝업 툴 호출. macro_view→show_macro_dashboard 식 1:1 매핑.
- **안전 우선**: risk_guardrail은 분류 단계에서 가장 먼저 판별. 경계 사례는 보수적으로 risk_guardrail에 귀속.
- **확장**: 운영하며 한 인텐트에 성격 다른 질문이 섞이면 그때 분할(예: 리포트 생성 진행상태가 필요해지면 별도 인텐트 추가).
- **주의**: 인텐트를 과도하게 잘게 쪼개면 분류기 혼동 + 데이터 부담 증가. 6개가 좋은 출발 균형.

### 보류 (5주 범위 밖 — 의도적으로 안 함)

- **16셀 체제 매트릭스**(버핏지수×공포탐욕 교차표): 우리 가중투표와 판정 철학이 다르다(교차표 vs 투표). 두 방식을 섞으면 "판정 출처 1곳" 원칙이 깨진다. 투표제 유지.
- **v3 리포트 스키마 전체**(재무건전성/가치함정/역발상/시나리오/사용자가설평가 등 10개 섹션, 28점 등급): 프로덕션급 깊이. 우리는 §6.5b의 6필드 소형 스키마로 충분. 등급 팩터→포지션 사이징→손절폭 자동 산출 체계도 "제안까지만" 원칙과 긴장 관계라 배제.
- **매크로 페이지 12섹션**(수급·환율·원자재·섹터히트맵·뉴스·반도체 사이클 등): 우리 대시보드는 국면 게이지 + 7지표 + 현금비중 3요소로 유지. 섹션 추가는 지표 추가가 아니라 화면 추가일 뿐이라 과제 가치 대비 비용이 크다.
- **증권사 컨센서스 PDF 요약, 백테스트 엔진, 주문 실행**: 범위 밖. 특히 주문 실행은 원칙 1에 의해 영구 배제.

### 우리가 이미 더 잘하고 있는 것

- **임계값 자동 생성**: 레퍼런스는 프롬프트 문자열과 코드 상수를 "수동으로 동일하게 유지 + 주석으로 경고"하는 방식(중복 허용). 우리는 `build_criteria_text()`로 코드에서 프롬프트를 **생성**하므로 사람이 3곳을 동시 수정할 필요가 없다. 이 강점은 유지하고, 스키마(3번째 층)만 추가한다.
- **판정의 LLM 완전 배제**: 레퍼런스는 GPT가 최종 등급을 부여(사전 계산값은 참고). 우리는 판정을 코드에 완전히 고정 — 과제의 안전 요건과 재현성 관점에서 우리 방식이 우월하다.
